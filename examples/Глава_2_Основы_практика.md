## 1. Значения, переменные и указатели

### Пример 1.1 — одно значение, несколько указателей

```rust
fn main() {
    let answer = 42;          // значение 42 типа i32, хранится в переменной (место на стеке)
    let p1 = &answer;         // p1 — значение-указатель (адрес answer)
    let p2 = &answer;         // p2 — ОТДЕЛЬНАЯ копия того же адреса

    // p1 и p2 — разные переменные, но хранят одинаковое значение (один адрес):
    println!("{:p} == {:p}", p1, p2);   // адреса совпадают
    assert_eq!(*p1, *p2);               // разыменование даёт одно значение

    // адреса самих переменных p1 и p2 — разные (это разные места на стеке):
    println!("{:p} != {:p}", &p1, &p2);
}
```

**Разбор.** Здесь важно различать три уровня: значение `42`, место (переменная `answer`), и значения-указатели. `p1` и `p2` хранят *копии* одного значения-указателя — менять одну переменную можно независимо от другой, как `var1`/`var2` в листинге 2-1.

### Пример 1.2 — где живёт строковый литерал

```rust
fn main() {
    let s1: &str = "Hello world";       // указатель в статическую память
    let s2: &str = "Hello world";       // тот же литерал — компилятор может слить их

    // сравнение значений (содержимого):
    assert_eq!(s1, s2);

    // сравнение указателей (адресов начала строки):
    println!("s1: {:p}, s2: {:p}", s1.as_ptr(), s2.as_ptr());

    let owned: String = String::from("Hello world"); // а это копия в КУЧЕ
    println!("heap: {:p}", owned.as_ptr());          // адрес заметно отличается
}
```

**Разбор.** Значение переменной `s1` — это «толстый указатель» (адрес + длина) на байты в статической памяти. `String` же владеет копией байтов в куче. Само *значение строки* («что имел в виду программист») одинаково во всех трёх случаях — различается лишь то, где лежит его представление.

## 2. Высокоуровневая модель: потоки (flows)

### Пример 2.1 — конфликт потоков и его устранение

```rust
fn main() {
    let mut x = 42;

    let r = &x;          // начало разделяемого потока от x
    println!("{r}");     // последнее использование r — поток заканчивается ЗДЕСЬ

    x += 1;              // эксклюзивный поток — ОК, разделяемый уже завершился
    println!("{x}");
}
```

А теперь сломаем:

```rust,compile_fail
fn main() {
    let mut x = 42;
    let r = &x;       // разделяемый поток начинается
    x += 1;           // ОШИБКА: эксклюзивный поток пересекается с живым разделяемым
    println!("{r}");  // ...потому что разделяемый поток дотягивается сюда
}
```

**Разбор.** Это NLL (non-lexical lifetimes) в действии: время жизни ссылки заканчивается на её последнем использовании, а не в конце области видимости. Первый вариант компилируется именно поэтому. Во втором варианте достаточно убрать последнюю строку — и код снова валиден.

### Пример 2.2 — затенение ≠ перезапись

```rust
fn main() {
    let x = String::from("first");
    let r = &x;                      // ссылка на ПЕРВУЮ x
    let x = String::from("second");  // затенение: новая переменная с тем же именем
    // первая x всё ещё жива (на неё ссылается r) — обе сосуществуют!
    println!("r = {r}, x = {x}");    // r = first, x = second
}
```

**Разбор.** Затенение не уничтожает старую переменную — к ней просто больше нельзя обратиться по имени. Существующие ссылки продолжают работать. Сравните с `x = String::from("second")` без `let`: это уже *присваивание*, оно сбрасывает старое значение и потребует `mut` (и конфликтовало бы с `r`).

## 3. Области памяти: стек, куча, статическая память

### Пример 3.1 — три области в одной программе

```rust
static GREETING: &str = "hi";          // статическая память

fn main() {
    let on_stack: [u64; 4] = [1, 2, 3, 4];      // стек: размер известен компилятору
    let on_heap: Box<[u64]> = vec![1, 2, 3, 4]  // куча: данные за указателем
        .into_boxed_slice();

    println!("static: {:p}", GREETING.as_ptr());
    println!("stack:  {:p}", &on_stack);
    println!("heap:   {:p}", on_heap.as_ptr());
}
```

**Разбор.** Запустите и сравните адреса — они окажутся в сильно разных диапазонах. На типичном Linux статическая память близка к началу адресного пространства программы, куча — выше, стек — у больших адресов и растёт вниз.

### Пример 3.2 — почему нельзя вернуть ссылку на локальную переменную

```rust,compile_fail
fn dangling() -> &i32 {
    let x = 42;
    &x   // ОШИБКА: x живёт в кадре dangling, кадр исчезнет при возврате
}
```

Рабочие альтернативы:

```rust
fn give_owned() -> i32 { 42 }                  // вернуть само значение (Copy)
fn give_boxed() -> Box<i32> { Box::new(42) }   // переместить в кучу — переживёт кадр
fn give_static() -> &'static i32 { &42 }       // константа попадает в статическую память
```

**Разбор.** Кадр стека освобождается при возврате — ссылка в него стала бы висячей. Куча и статическая память от кадров не зависят, поэтому `Box` и `&'static` работают.

### Пример 3.3 — `Box::leak` для конфигурации на всю программу

```rust
#[derive(Debug)]
struct Config {
    verbose: bool,
    name: String,
}

fn load_config() -> &'static Config {
    let cfg = Config { verbose: true, name: "demo".into() };
    // Намеренная утечка: память не освободится до конца программы,
    // зато получаем 'static-ссылку, которую можно раздавать куда угодно
    // (в том числе в потоки) без Arc и без клонирования.
    Box::leak(Box::new(cfg))
}

fn main() {
    let cfg: &'static Config = load_config();
    let handle = std::thread::spawn(move || {
        // поток может пережить main-кадр — но cfg это 'static, всё безопасно
        println!("worker sees: {cfg:?}");
    });
    handle.join().unwrap();
}
```

### Пример 3.4 — `const` против `static`

```rust
const LIMIT: i32 = 10;        // не место: подставляется в код при компиляции
static COUNTER_START: i32 = 10;  // место: одна локация в статической памяти

fn main() {
    let a = &LIMIT;           // создаётся ВРЕМЕННОЕ значение, ссылка на него
    let b = &LIMIT;           // другое временное — адреса могут различаться
    let c = &COUNTER_START;   // всегда один и тот же адрес
    let d = &COUNTER_START;
    println!("const:  {:p} vs {:p}", a, b);  // могут отличаться
    println!("static: {:p} vs {:p}", c, d);  // всегда равны
    assert!(std::ptr::eq(c, d));
}
```

**Разбор.** `const` — имя для значения, у него нет своей памяти; каждое `&LIMIT` может материализовать новую временную копию. `static` — настоящее место с единственным адресом на всю программу.

## 4. Владение: перемещение, копирование, сброс

### Пример 4.1 — move в функции и обратно

```rust
fn consume(s: String) -> String {
    println!("consumed: {s}");
    s   // возвращаем владение обратно
}

fn main() {
    let s = String::from("hello");
    let s = consume(s);   // s перемещён в функцию и возвращён (с затенением)
    println!("still mine: {s}");
}
```

**Разбор.** Передача по значению — это перемещение. «Вернуть владение» — рабочий, но громоздкий паттерн; в разделе 5 увидим, что обычно вместо этого одалживают через `&`/`&mut`.

### Пример 4.2 — `Copy` против `Clone` на собственном типе

```rust
#[derive(Debug, Clone, Copy)]   // Copy возможен: оба поля — Copy
struct Point { x: i32, y: i32 }

#[derive(Debug, Clone)]         // Copy НЕВОЗМОЖЕН: String владеет памятью в куче
struct Label { text: String }

fn main() {
    let p1 = Point { x: 1, y: 2 };
    let p2 = p1;                 // копирование битов, p1 жив
    println!("{p1:?} {p2:?}");

    let l1 = Label { text: "a".into() };
    let l2 = l1.clone();         // явное глубокое копирование
    let l3 = l1;                 // перемещение, l1 мёртв
    println!("{l2:?} {l3:?}");
    // println!("{l1:?}");       // ОШИБКА: value borrowed here after move
}
```

**Разбор.** Попробуйте добавить `Copy` к `Label` — компилятор откажет: `String` не `Copy`, потому что побитовая копия создала бы два владельца одной кучи (тот самый double free из главы).

### Пример 4.3 — наблюдаем порядок сброса

```rust
struct Noisy(&'static str);

impl Drop for Noisy {
    fn drop(&mut self) {
        println!("drop {}", self.0);
    }
}

fn main() {
    let _a = Noisy("a");                       // переменные: обратный порядок
    let _b = Noisy("b");
    let _tuple = (Noisy("t1"), Noisy("t2"));   // вложенные: прямой порядок
    println!("end of main");
}
// Вывод:
// end of main
// drop t1      <- кортеж (объявлен последним) сбрасывается первым,
// drop t2      <- но ВНУТРИ кортежа порядок прямой: t1, затем t2
// drop b       <- переменные в обратном порядке
// drop a
```

**Разбор.** Это иллюстрация врезки «Порядок сброса»: переменные — в обратном порядке (поздние могут ссылаться на ранние), вложенные значения — в порядке исходного кода.

### Пример 4.4 — ловушка: `let _ =` против `let _x =`

```rust
struct Guard;
impl Drop for Guard {
    fn drop(&mut self) { println!("guard released"); }
}

fn main() {
    let _x = Guard;   // привязка: живёт до конца области видимости
    let _ = Guard;    // НЕ привязка: значение сбрасывается НЕМЕДЛЕННО
    println!("work happens here");
}
// Вывод:
// guard released        <- от `let _ =`, до начала работы!
// work happens here
// guard released        <- от `let _x =`, в конце области
```

**Разбор.** `_` — не переменная, а шаблон «игнорировать»: значение не получает владельца и сбрасывается сразу. Для RAII-объектов (блокировки, таймеры) это классическая ошибка.

## 5. Заимствование: разделяемые и изменяемые ссылки

### Пример 5.1 — правило «много читателей ИЛИ один писатель»

```rust
fn main() {
    let mut data = vec![1, 2, 3];

    // сколько угодно разделяемых ссылок одновременно:
    let r1 = &data;
    let r2 = &data;
    println!("{r1:?} {r2:?}");           // оба потока заканчиваются здесь

    // теперь можно взять эксклюзивную:
    let w = &mut data;
    w.push(4);
    println!("{w:?}");                   // эксклюзивный поток заканчивается

    println!("{data:?}");                // и снова доступен владелец
}
```

### Пример 5.2 — почему эксклюзивность спасает от ошибок: инвалидация итератора

```rust,compile_fail
fn main() {
    let mut v = vec![1, 2, 3];
    for x in &v {            // разделяемое заимствование v на весь цикл
        if *x == 2 {
            v.push(4);       // ОШИБКА: &mut при живом & — а ведь push мог бы
        }                    // перевыделить буфер и сделать x висячим!
    }
}
```

**Разбор.** В C++ аналогичный код компилируется и приводит к UB (итератор инвалидируется при реаллокации). Правило эксклюзивности — не бюрократия, а защита от целого класса багов use-after-free.

### Пример 5.3 — `mem::take`, `mem::replace`, `mem::swap` на практике

```rust
#[derive(Debug, Default)]
struct Connection {
    buffer: Vec<u8>,
}

impl Connection {
    /// Забрать накопленный буфер, оставив на его месте пустой.
    /// Через &mut нельзя ПРОСТО вынести Vec — нужно что-то оставить взамен.
    fn drain_buffer(&mut self) -> Vec<u8> {
        std::mem::take(&mut self.buffer)
        // эквивалент: std::mem::replace(&mut self.buffer, Vec::new())
    }
}

fn main() {
    let mut conn = Connection { buffer: vec![1, 2, 3] };
    let data = conn.drain_buffer();
    println!("got {data:?}, left {:?}", conn.buffer);  // got [1,2,3], left []

    // swap: обменять содержимое двух мест без владения
    let mut a = String::from("a");
    let mut b = String::from("b");
    std::mem::swap(&mut a, &mut b);
    println!("{a} {b}");   // b a
}
```

**Разбор.** Это паттерн из листинга 2-7 в прикладном виде. `take` незаменим в state-машинах: `let old_state = mem::take(&mut self.state)` позволяет владеть старым состоянием при вычислении нового.

### Пример 5.4 — изменяемость применяется к непосредственной памяти (двойные ссылки)

```rust
fn main() {
    let x = 10;
    let y = 20;

    let mut r: &i32 = &x;     // r можно ПЕРЕНАЦЕЛИВАТЬ, но не менять *r
    r = &y;                   // ок
    // *r = 30;               // ОШИБКА: за &i32 менять нельзя

    let mut a = 10;
    let r2: &mut i32 = &mut a; // *r2 менять МОЖНО, но r2 не перенацелить
    *r2 = 30;                  // ок
    // r2 = &mut другое;       // ОШИБКА: r2 не объявлен как mut

    // и комбинация из листинга 2-6:
    let z: &mut &i32 = &mut r; // через z можно поменять, КУДА указывает r
    *z = &x;                   // ок: меняем значение-указатель r
    // **z = 99;               // ОШИБКА: конечный i32 за & — неизменяем
}
```

**Разбор.** Запомните мнемонику: `mut` слева от имени (`let mut r`) — можно переприсваивать саму переменную; `&mut` в типе — можно менять то, на что указываем. Это независимые свойства.

## 6. Внутренняя изменяемость: `Cell`, `RefCell`, `Mutex`, атомики

### Пример 6.1 — `Cell`: замена значения через `&`

```rust
use std::cell::Cell;

struct Counter {
    hits: Cell<u32>,   // хотим считать вызовы в методе с &self
}

impl Counter {
    fn record(&self) {                       // ВНИМАНИЕ: &self, не &mut self
        self.hits.set(self.hits.get() + 1);  // get копирует, set заменяет
    }
}

fn main() {
    let c = Counter { hits: Cell::new(0) };
    let r1 = &c;
    let r2 = &c;        // сколько угодно разделяемых ссылок
    r1.record();
    r2.record();
    println!("{}", c.hits.get());   // 2
}
```

**Разбор.** `Cell` никогда не выдаёт ссылку внутрь себя — только копирует (`get`) или заменяет (`set`/`replace`/`take`). Поэтому изменение через `&` безопасно: висячей ссылки на внутренности возникнуть не может. Цена: работает удобно только с `Copy`-типами и только в одном потоке (`Cell` не `Sync`).

### Пример 6.2 — `RefCell`: заимствование, проверяемое в рантайме

```rust
use std::cell::RefCell;

fn main() {
    let log = RefCell::new(Vec::<String>::new());

    log.borrow_mut().push("first".into());    // ~&mut, проверка в рантайме
    log.borrow_mut().push("second".into());

    {
        let read = log.borrow();              // ~&, тоже в рантайме
        println!("{} entries", read.len());

        // а вот это ПАНИКА (раскомментируйте):
        // log.borrow_mut().push("boom".into());
        // already borrowed: BorrowMutError
    } // read сброшен — счётчик заимствований обнулился

    log.borrow_mut().push("third".into());    // снова можно
}
```

**Разбор.** `RefCell` переносит правило «много `&` ИЛИ один `&mut`» из компиляции в рантайм: нарушение — не UB, а паника. Используйте, когда корректность алиасинга вы доказали сами, а компилятор статически доказать не может (графы, колбэки, кэши).

### Пример 6.3 — `Mutex` и атомики: внутренняя изменяемость между потоками

```rust
use std::sync::atomic::{AtomicUsize, Ordering};
use std::sync::Mutex;
use std::thread;

static FAST_COUNTER: AtomicUsize = AtomicUsize::new(0);   // без блокировки
static SLOW_LOG: Mutex<Vec<usize>> = Mutex::new(Vec::new()); // с блокировкой

fn main() {
    thread::scope(|s| {
        for i in 0..4 {
            s.spawn(move || {
                FAST_COUNTER.fetch_add(1, Ordering::Relaxed); // изменение через &static!
                SLOW_LOG.lock().unwrap().push(i);             // lock даёт ~&mut Vec
            });
        }
    });
    println!("count = {}", FAST_COUNTER.load(Ordering::Relaxed)); // 4
    println!("log = {:?}", SLOW_LOG.lock().unwrap());
}
```

**Разбор.** Обе статики изменяются через разделяемую ссылку — это и есть внутренняя изменяемость. `AtomicUsize` (категория «заменить значение») опирается на атомарные инструкции CPU; `Mutex` (категория «выдать `&mut`») гарантирует эксклюзивность блокировкой. Под капотом у обоих — `UnsafeCell`.

### Пример 7.3 — обобщённое время жизни в типе + почему важен `Drop`

```rust
struct Parser<'doc> {
    rest: &'doc str,            // ссылка на ещё не разобранную часть документа
}

impl<'doc> Parser<'doc> {
    fn new(doc: &'doc str) -> Self {
        Parser { rest: doc }
    }

    /// Возвращаемая ссылка живёт столько же, сколько ДОКУМЕНТ ('doc),
    /// а не сколько &mut self — это и есть смысл обобщённого времени жизни.
    fn next_word(&mut self) -> Option<&'doc str> {
        let rest = self.rest.trim_start();
        if rest.is_empty() { return None; }
        let end = rest.find(' ').unwrap_or(rest.len());
        let (word, tail) = rest.split_at(end);
        self.rest = tail;
        Some(word)
    }
}

fn main() {
    let doc = String::from("alpha beta gamma");
    let first;
    {
        let mut p = Parser::new(&doc);
        first = p.next_word().unwrap();
    }                       // p сброшен, но first жив: он привязан к doc, не к p!
    println!("{first}");    // alpha
}
```

**Разбор.** Если бы `next_word` возвращал `Option<&str>` с временем жизни от `&mut self` (так выведет элизия для `fn next_word(&mut self) -> Option<&str>` — проверьте!), `first` не пережил бы `p`, и каждый следующий вызов конфликтовал бы с ещё живым `first`. Возврат `&'doc str` отвязывает результат от парсера. Это та же идея, что в `StrSplit` из листинга 2-10.

### Пример 7.4 — когда одного времени жизни мало

```rust
/// Вернуть кусок `haystack` до первого вхождения `needle`.
/// Результат должен зависеть ТОЛЬКО от 'h — иначе не написать find_in_temp.
fn before<'h>(haystack: &'h str, needle: &str) -> &'h str {
    match haystack.find(needle) {
        Some(i) => &haystack[..i],
        None => haystack,
    }
}

fn find_in_temp(s: &str) -> &str {
    let sep = String::from(",");  // временная строка, умрёт при возврате
    before(s, &sep)               // ок: результат не привязан к sep
}

fn main() {
    println!("{}", find_in_temp("a,b,c"));   // a
}
```

**Разбор.** Объявите `before` как `fn before<'a>(haystack: &'a str, needle: &'a str) -> &'a str` — и `find_in_temp` перестанет компилироваться: результат окажется привязан к минимуму из двух времён жизни, включая локальную `sep`. Это в точности проблема `str_before` из листинга 2-10.

## 8. Вариантность

### Пример 8.1 — ковариантность `&'a T`: длинное подставляется вместо короткого

```rust
fn pick_first<'a>(a: &'a str, b: &'a str) -> &'a str {
    if !a.is_empty() { a } else { b }
}

fn main() {
    let long: &'static str = "static string";
    let s = String::from("short-lived");
    let short: &str = &s;

    // 'static-ссылка спокойно «ужимается» до времени жизни short:
    let result = pick_first(long, short);
    println!("{result}");
}
```

**Разбор.** Компилятор унифицирует оба аргумента к одному `'a`. Это возможно лишь потому, что `&'a T` ковариантен по `'a`: `&'static str <: &'a str`. Будь ссылки инвариантны, такая функция была бы почти бесполезна.

### Пример 8.2 — инвариантность `&mut T`: почему «ужимать» нельзя

```rust,compile_fail
fn put_short<'a>(v: &mut Vec<&'a str>, s: &'a str) {
    v.push(s);
}

fn main() {
    let mut v: Vec<&'static str> = vec!["forever"];
    let temp = String::from("temporary");
    // ОШИБКА: &mut Vec<&'static str> нельзя передать как &mut Vec<&'a str>,
    // иначе мы бы запихнули короткоживущую ссылку в "вечный" вектор:
    put_short(&mut v, &temp);
    drop(temp);
    println!("{}", v[1]);   // ...и читали бы висячую ссылку!
}
```

**Разбор.** Это дословная реализация атаки из главы: если бы `&mut T` был ковариантен по `T`, последняя строка читала бы освобождённую память. Инвариантность `&mut T` по `T` закрывает дыру на этапе компиляции.

### Пример 8.3 — `MutStr` из листинга 2-11, обе версии рядом

```rust
// Версия с двумя временами жизни — КОМПИЛИРУЕТСЯ:
struct MutStr<'a, 'b> {
    s: &'a mut &'b str,
}

fn main() {
    let mut s = "hello";
    *MutStr { s: &mut s }.s = "world";
    println!("{s}");   // world
}
```

```rust,compile_fail
// Версия с одним временем жизни — НЕ компилируется:
struct MutStrOne<'a> {
    s: &'a mut &'a str,    // &mut инвариантен по &'a str
}

fn main() {
    let mut s = "hello";
    *MutStrOne { s: &mut s }.s = "world";
    println!("{s}");
    // ошибка: cannot borrow `s` as immutable because it is
    // also borrowed as mutable — заимствование "залипло" до 'static
}
```

**Разбор.** В одно-лайфтаймовой версии `'a` выводится в `'static` (из-за `"hello"`), а инвариантность `&mut` по `&'a str` не даёт его сократить — изменяемое заимствование `s` растягивается до конца программы. Два независимых времени жизни позволяют `'a` (заимствованию) закончиться сразу после `*... = "world"`, а `'b` остаться `'static`.

### Пример 8.4 — таблица вариантности (для самопроверки)

| Тип | по `'a` | по `T` |

## Шпаргалка по главе

- **Значение** — тип + элемент его области значений; **место** — где лежит представление; **переменная** — именованное место на стеке; **указатель** — значение-адрес места.
- **Стек** — кадры функций, освобождаются при возврате; **куча** — живёт до явного освобождения (`Box`, сброс); **статическая память** — живёт всю программу (`static`, литералы, код).
- `'static` как ограничение (`T: 'static`) = «не заимствует ничего короткоживущего», а не «лежит в статической памяти».
- **Владелец один**; перемещение передаёт владение; `Copy` — побитовое дублирование, только для типов без ресурсов; сброс автоматический: переменные — в обратном порядке, вложенные значения — в прямом.
- **`&T`** — много, значение не меняется (компилятор на это рассчитывает!); **`&mut T`** — одна, гарантированно без алиасов; вынести значение из-за `&mut` можно только оставив замену (`take`/`replace`/`swap`).
- **Внутренняя изменяемость**: `Mutex`/`RefCell` выдают `&mut` под контролем; `Cell`/атомики заменяют значение, не выдавая ссылок; внутри всех — `UnsafeCell`.
- **Время жизни** — область кода, где ссылка обязана быть действительной: заканчивается на последнем использовании (NLL), может иметь дыры, не равно области видимости.
- **Вариантность**: `&'a T` ковариантен по `'a` и `T`; `&mut T` инвариантен по `T`; `fn(T)` контравариантен по `T`. Инвариантность за `&mut` — причина, по которой иногда нужны два параметра времени жизни.
