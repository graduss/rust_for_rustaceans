## 1. Выравнивание и раскладка

### Пример 1.1 — листинг 3-1 в цифрах: `repr(C)` против `repr(Rust)`

```rust
#[repr(C)]
struct FooC {
    tiny: bool,    // 1 байт + 3 байта набивки (для normal)
    normal: u32,   // 4 байта
    small: u8,     // 1 байт + 7 байт набивки (для long)
    long: u64,     // 8 байт
    short: u16,    // 2 байта + 6 байт финальной набивки (до кратности 8)
}

struct FooRust {   // repr(Rust): компилятор волен переупорядочить поля
    tiny: bool,
    normal: u32,
    small: u8,
    long: u64,
    short: u16,
}

fn main() {
    use std::mem::{align_of, size_of};
    println!("FooC:    size {:2}, align {}", size_of::<FooC>(), align_of::<FooC>());
    println!("FooRust: size {:2}, align {}", size_of::<FooRust>(), align_of::<FooRust>());
    // FooC:    size 32, align 8   <- 16 байт полей + 16 байт набивки!
    // FooRust: size 16, align 8   <- поля переставлены по убыванию размера
}
```

**Разбор.** Это арифметика из главы, проверенная компилятором. `repr(C)` обязан класть поля в порядке объявления — половина структуры уходит на набивку. Раскладка Rust по умолчанию переупорядочивает поля (`long`, `normal`, `short`, `small`, `tiny`) и набивка не нужна вовсе. Помните: размер `FooRust` = 16 — это поведение *текущего* компилятора, а не гарантия.

### Пример 1.2 — `repr(packed)` и `repr(align(n))`

```rust
#[repr(C, packed)]
struct Packed {        // никакой набивки: 1+4+1+8+2 = 16 байт, выравнивание 1
    tiny: bool,
    normal: u32,
    small: u8,
    long: u64,
    short: u16,
}

#[repr(align(64))]     // принудительно на отдельную линию кэша
struct CacheLinePadded {
    counter: u64,      // 8 байт данных + 56 байт набивки
}

fn main() {
    use std::mem::{align_of, size_of};
    assert_eq!((size_of::<Packed>(), align_of::<Packed>()), (16, 1));
    assert_eq!((size_of::<CacheLinePadded>(), align_of::<CacheLinePadded>()), (64, 64));

    // ЛОВУШКА packed: ссылка на невыровненное поле — ошибка компиляции,
    // потому что &u64 обязан быть выровнен по 8:
    let p = Packed { tiny: true, normal: 1, small: 2, long: 3, short: 4 };
    // let r = &p.long;          // ОШИБКА E0793: reference to packed field is unaligned
    let copied = p.long;         // а копирование значения — можно
    println!("{copied}");
}
```

**Разбор.** `packed` экономит память ценой невыровненных обращений, а `align(64)` тратит память ради того, чтобы соседние счётчики в массиве не делили линию кэша (защита от ложного разделения). Запрет `&p.long` — не каприз: безопасный код вправе считать, что любая `&u64` выровнена по 8; `packed` это сломал бы.

### Пример 1.3 — `repr(transparent)` для newtype

```rust
#[repr(transparent)]
struct Meters(f64);     // ГАРАНТИРОВАННО та же раскладка, что у f64

fn main() {
    assert_eq!(std::mem::size_of::<Meters>(), std::mem::size_of::<f64>());
    assert_eq!(std::mem::align_of::<Meters>(), std::mem::align_of::<f64>());
    // Благодаря transparent безопасно (в unsafe-контекстах и FFI)
    // трактовать &Meters как &f64 и наоборот.
}
```

**Разбор.** Без `repr(transparent)` совпадение раскладок `Meters` и `f64` — счастливая случайность, на которую нельзя опираться. С ним — контракт, на котором строятся FFI-обёртки и приёмы вроде `&Vec<Meters> -> &Vec<f64>` (через unsafe, глава 10).

## 2. Сложные типы: перечисления и дискриминант

### Пример 2.1 — где живёт дискриминант

```rust
enum Shape {
    Circle { radius: f64 },                 // 8 байт полезной нагрузки
    Rect { w: f64, h: f64 },                // 16 байт полезной нагрузки
}

fn main() {
    use std::mem::size_of;
    // Раскладка ~ как у union вариантов + скрытое поле дискриминанта:
    // max(8, 16) = 16 байт нагрузки + дискриминант, выровненный до 8 => 24
    println!("Shape: {}", size_of::<Shape>());     // 24
    println!("дискриминант + набивка: {}", size_of::<Shape>() - 16);
}
```

### Пример 2.2 — нишевая оптимизация: `Option`, который ничего не стоит

```rust
use std::num::NonZeroU32;

fn main() {
    use std::mem::size_of;
    // Указатели не бывают нулевыми => Null годится как дискриминант None:
    assert_eq!(size_of::<Option<Box<u8>>>(), size_of::<Box<u8>>());   // 8 == 8
    assert_eq!(size_of::<Option<&u8>>(),     size_of::<&u8>());       // 8 == 8

    // bool занимает байт, но допустимы только 0 и 1 => 2..=255 — ниша:
    assert_eq!(size_of::<Option<bool>>(), 1);

    // NonZeroU32 объявляет 0 невозможным => Option бесплатен:
    assert_eq!(size_of::<Option<NonZeroU32>>(), 4);

    // А вот u32 использует все 2^32 битовых комбинаций — нужен лишний разряд,
    // который из-за выравнивания раздувается до 4 байт:
    assert_eq!(size_of::<Option<u32>>(), 8);
}
```

**Разбор.** Компилятор Rust пользуется свободой `repr(Rust)`: если в типе есть *ниша* (битовая комбинация, которая никогда не является допустимым значением), дискриминант enum прячется в неё. Поэтому `Option<&T>`, `Option<Box<T>>`, `Result<(), NonZeroU32>` не больше своей полезной нагрузки. Это одна из причин, почему «нулевые указатели» в Rust не нужны: `Option<&T>` так же дёшев, но безопасен.

## 3. DST и толстые указатели

### Пример 3.1 — измеряем толщину указателей

```rust
trait Draw { fn draw(&self) -> String; }
impl Draw for i32 { fn draw(&self) -> String { format!("int {self}") } }

fn main() {
    use std::mem::size_of;
    const WORD: usize = size_of::<usize>();

    // Тонкие указатели — одно слово:
    assert_eq!(size_of::<&u8>(), WORD);
    assert_eq!(size_of::<Box<[u8; 4]>>(), WORD);   // [u8; 4] — Sized!

    // Толстые указатели — два слова:
    assert_eq!(size_of::<&[u8]>(), 2 * WORD);       // указатель + длина
    assert_eq!(size_of::<&str>(), 2 * WORD);        // str — тоже срез (байтов)
    assert_eq!(size_of::<&dyn Draw>(), 2 * WORD);   // указатель + vtable
    assert_eq!(size_of::<Box<dyn Draw>>(), 2 * WORD);

    // Option<толстый указатель> — ниша работает и тут:
    assert_eq!(size_of::<Option<&[u8]>>(), 2 * WORD);
}
```

### Пример 3.2 — один и тот же массив, разные «дополнения»

```rust
fn main() {
    let arr: [u16; 4] = [10, 20, 30, 40];   // Sized: размер известен компилятору

    let whole: &[u16] = &arr;       // толстый указатель: (адрес, len=4)
    let part: &[u16] = &arr[1..3];  // тот же буфер, но (адрес+2, len=2)

    println!("{whole:?} / {part:?}");        // [10,20,30,40] / [20,30]
    assert_eq!(whole.len(), 4);
    assert_eq!(part.len(), 2);
    // len() не лезет в память массива — длина лежит В САМОМ указателе.
}
```

### Пример 3.3 — `?Sized`: принимаем и `str`, и `dyn Trait` по ссылке

```rust
use std::fmt::Display;

// По умолчанию T: Sized — этой функции нельзя передать str или dyn Display:
fn show_sized<T: Display>(v: &T) -> String { format!("[{v}]") }

// ?Sized снимает ограничение — теперь T может быть DST:
fn show_any<T: Display + ?Sized>(v: &T) -> String { format!("[{v}]") }

fn main() {
    let s: &str = "hi";
    let d: &dyn Display = &42;

    // show_sized(s);   // ОШИБКА: str не Sized (T выводится в str, не &str)
    // show_sized(d);   // ОШИБКА: dyn Display не Sized
    println!("{}", show_any(s));   // [hi]
    println!("{}", show_any(d));   // [42]
    println!("{}", show_any(&7));  // [7] — Sized-типы тоже подходят
}
```

**Разбор.** Ограничение `T: Sized` дописывается к каждому обобщённому параметру неявно. Сам DST нельзя положить в переменную или передать по значению, но *за ссылкой* (толстым указателем) — пожалуйста. Именно поэтому в сигнатурах стандартной библиотеки так часто встречается `T: ?Sized` (`Box<T>`, `Arc<T>`, `Borrow<T>`…) — это расширяет, а не сужает множество допустимых типов.

## 4. Статическая и динамическая диспетчеризация

### Пример 4.1 — обе диспетчеризации бок о бок

```rust
trait Greet {
    fn hello(&self) -> String;
}

struct En;
struct Ru;
impl Greet for En { fn hello(&self) -> String { "hello".into() } }
impl Greet for Ru { fn hello(&self) -> String { "привет".into() } }

// Статическая: мономорфизация создаст greet_static::<En> и greet_static::<Ru>,
// в каждой копии адрес hello известен на этапе компиляции (возможен inline):
fn greet_static(g: &impl Greet) -> String {
    g.hello()
}

// Динамическая: ОДНА копия функции; адрес hello достаётся из vtable в рантайме:
fn greet_dyn(g: &dyn Greet) -> String {
    g.hello()
}

fn main() {
    println!("{} {}", greet_static(&En), greet_static(&Ru));
    println!("{} {}", greet_dyn(&En), greet_dyn(&Ru));

    // Главная сила dyn — ГЕТЕРОГЕННЫЕ коллекции:
    let all: Vec<Box<dyn Greet>> = vec![Box::new(En), Box::new(Ru)];
    for g in &all {
        println!("{}", g.hello());
    }
    // Vec<impl Greet> такого не может: один Vec — один конкретный T.
}
```

**Разбор.** `&impl Greet` — сахар для `<T: Greet>(g: &T)`: копия кода на каждый тип, статический адрес, полные оптимизации, но больше машинного кода и дольше компиляция. `&dyn Greet` — одно тело функции на все типы, вызов через vtable. Заметьте: чтобы превратить `&En` в `&dyn Greet`, не нужно ничего писать — компилятор сам строит толстый указатель.

### Пример 4.2 — объектная безопасность: что нельзя сделать `dyn`

```rust,compile_fail
fn broken(c: &dyn Clone) -> () {}
// ОШИБКА: `Clone` is not dyn compatible:
// clone() возвращает Self, а у dyn Clone "Self" неизвестен компилятору.
```

```rust
// Приём из главы: `where Self: Sized` исключает метод из dyn-интерфейса.
trait Stack {
    fn push(&mut self, v: i32);
    fn pop(&mut self) -> Option<i32>;

    // Метод возвращает Self => сделал бы трейт объектно-небезопасным.
    // Помечаем его Sized — он "невидим" через dyn Stack, и трейт спасён:
    fn duplicate(&self) -> Self
    where
        Self: Sized,
        Self: Clone,
    {
        self.clone()
    }
}

#[derive(Clone)]
struct VecStack(Vec<i32>);
impl Stack for VecStack {
    fn push(&mut self, v: i32) { self.0.push(v); }
    fn pop(&mut self) -> Option<i32> { self.0.pop() }
}

fn main() {
    let mut s: Box<dyn Stack> = Box::new(VecStack(vec![]));
    s.push(1);
    s.push(2);
    assert_eq!(s.pop(), Some(2));     // обычные методы доступны через dyn
    // s.duplicate();                 // ОШИБКА: метод требует Self: Sized

    let concrete = VecStack(vec![7]);
    let _copy = concrete.duplicate(); // а на конкретном типе — пожалуйста
}
```

**Разбор.** В vtable должна попасть одна конкретная функция на метод. Методы, возвращающие `Self`, обобщённые методы и статические методы (без `self`) этого не позволяют. `where Self: Sized` на методе — стандартный приём «вырезать» проблемный метод из dyn-интерфейса, не теряя его для конкретных типов (так в std устроены `Iterator::map`, `collect` и прочие).

### Пример 4.3 — необобщённая внутренняя функция (из врезки)

```rust
use std::fmt::Display;

// Наивно: ВСЁ тело копируется для каждого T (String, i32, f64, ...)
fn render_naive<T: Display>(value: T) -> String {
    let s = value.to_string();
    // ...представьте здесь 200 строк форматирования, не зависящих от T...
    format!(">>> {} <<<", s.trim())
}

// Лучше: от T зависит только to_string; остальное — одна разделяемая копия:
fn render_smart<T: Display>(value: T) -> String {
    fn inner(s: String) -> String {              // НЕ обобщённая
        // ...те же 200 строк компилируются ОДИН раз...
        format!(">>> {} <<<", s.trim())
    }
    inner(value.to_string())                     // зависящая от T часть — минимальна
}

fn main() {
    assert_eq!(render_naive(" 42 "), render_smart(" 42 "));
    assert_eq!(render_smart("hi"), ">>> hi <<<");
}
```

**Разбор.** Так в std устроены, например, `std::fs::write` и `Path::join`: публичная обобщённая обёртка (`impl AsRef<Path>`) сразу деградирует к необобщённой внутренней функции. Мономорфизируется только тонкий слой конверсии — бинарник меньше, компиляция быстрее, кэш инструкций счастливее.

## 5. Обобщённые трейты: параметр типа против связанного типа

### Пример 5.1 — связанный тип: одна реализация на тип

```rust
trait Measure {
    type Unit;                      // диктуется реализацией, выбора у пользователя нет
    fn measure(&self) -> Self::Unit;
}

struct Road { km: f64 }

impl Measure for Road {
    type Unit = f64;
    fn measure(&self) -> f64 { self.km }
}

// Вторая реализация для Road НЕВОЗМОЖНА — E0119 conflicting implementations:
// impl Measure for Road { type Unit = u32; ... }

fn main() {
    let r = Road { km: 3.5 };
    // Никакой неоднозначности: Unit однозначно следует из типа Road.
    let d = r.measure();
    println!("{d} km");
}
```

### Пример 5.2 — параметр типа: сколько угодно реализаций

```rust
trait ConvertTo<T> {
    fn convert(&self) -> T;
}

struct Celsius(f64);

impl ConvertTo<f64> for Celsius {            // в кельвины
    fn convert(&self) -> f64 { self.0 + 273.15 }
}
impl ConvertTo<String> for Celsius {         // в строку
    fn convert(&self) -> String { format!("{}°C", self.0) }
}

fn main() {
    let t = Celsius(20.0);
    // ЦЕНА гибкости: теперь нужно подсказывать, КАКУЮ реализацию мы хотим:
    let k: f64 = t.convert();
    let s: String = t.convert();
    // или через turbofish-подобный синтаксис уточнения:
    let s2 = ConvertTo::<String>::convert(&t);
    println!("{k} {s} {s2}");
}
```

**Разбор.** Эмпирическое правило главы в действии. `Measure` со связанным типом: ноль аннотаций при использовании, но ровно одна реализация на тип (как `Iterator::Item`, `Deref::Target`). `ConvertTo<T>` с параметром: реализаций сколько угодно (как `From<T>`, `PartialEq<Rhs>`), но пользователю приходится разрешать неоднозначность аннотациями. ## 6. Когерентность и правило сироты

### Пример 6.1 — что можно и что нельзя

```rust
use std::fmt;

struct Wrapper(bool);             // наш локальный тип
trait Loud { fn shout(&self) -> String; }   // наш локальный трейт

// СВОЙ трейт для ЧУЖОГО типа — можно:
impl Loud for bool {
    fn shout(&self) -> String { format!("{}!!!", self).to_uppercase() }
}

// ЧУЖОЙ трейт для СВОЕГО типа — можно:
impl fmt::Display for Wrapper {
    fn fmt(&self, f: &mut fmt::Formatter) -> fmt::Result {
        write!(f, "Wrapper({})", self.0)
    }
}

// Исключение для покрытых реализаций: чужой трейт (From) для чужого типа (Vec),
// но наш Wrapper стоит в параметрах ДО любых обобщённых — можно:
impl<T: Default> From<Wrapper> for Vec<T> {
    fn from(_: Wrapper) -> Self { vec![T::default()] }
}

fn main() {
    println!("{}", true.shout());            // TRUE!!!
    println!("{}", Wrapper(false));          // Wrapper(false)
    let v: Vec<i32> = Wrapper(true).into();
    println!("{v:?}");                       // [0]
}
```

А вот это — классическое нарушение правила сироты:

```rust,compile_fail
use std::fmt;
// ЧУЖОЙ трейт (Display из std) для ЧУЖОГО типа (bool из std):
impl fmt::Display for bool {                 // ОШИБКА E0117:
    fn fmt(&self, f: &mut fmt::Formatter) -> fmt::Result {
        write!(f, "да/нет")
    }
}
// only traits defined in the current crate can be implemented
// for primitive types / arbitrary types
fn main() {}
```

**Разбор.** Когерентность: для пары (тип, трейт) во всей программе должна существовать максимум одна реализация. Правило сироты гарантирует это локально: хотя бы один из двух — трейт или тип — должен быть вашим. `impl From<Wrapper> for Vec<T>` проходит по исключению «покрытых реализаций»: локальный `Wrapper` стоит первым из конкретных типов в `From<...> for ...`.

### Пример 6.2 — паттерн newtype: легальный обход правила сироты

```rust
use std::fmt;

// Хотим "Display для Vec<u8> в hex" — нельзя (чужой трейт, чужой тип).
// Решение: завернуть чужой тип в свой:
struct Hex(Vec<u8>);

impl fmt::Display for Hex {
    fn fmt(&self, f: &mut fmt::Formatter) -> fmt::Result {
        for b in &self.0 {
            write!(f, "{b:02x}")?;
        }
        Ok(())
    }
}

fn main() {
    println!("{}", Hex(vec![0xDE, 0xAD, 0xBE, 0xEF]));   // deadbeef
}
```

**Разбор.** Newtype — стандартный ответ на правило сироты: `Hex` локален, значит `Display for Hex` легален. В связке с `#[repr(transparent)]` и `Deref` обёртка может быть практически бесплатной и прозрачной. Это же спасает от неоднозначности: будь `Display for Vec<u8>` разрешён всем, два крейта с разными форматами не смогли бы сосуществовать.

### Пример 6.3 — почему бланкетная реализация — это серьёзное обязательство

```rust
trait Describe {
    fn describe(&self) -> String;
}

// Бланкетная реализация: один impl покрывает БЕСКОНЕЧНО много типов.
impl<T: std::fmt::Debug> Describe for T {
    fn describe(&self) -> String { format!("{self:?}") }
}

struct Custom;                   // ВАЖНО: Custom НЕ реализует Debug

// Неожиданно, но это КОМПИЛИРУЕТСЯ: компилятор доказывает непересечение,
// потому что Custom: Debug не выполняется, а нижестоящие крейты не могут
// реализовать чужой Debug для чужого Custom (само правило сироты!):
impl Describe for Custom {
    fn describe(&self) -> String { "custom".into() }
}

fn main() {
    println!("{}", 42.describe());            // через бланкетную
    println!("{}", vec!["a", "b"].describe()); // через бланкетную
    println!("{}", Custom.describe());         // через специфичную
}
```

А вот когда пересечение реально — E0119:

```rust,compile_fail
trait Describe { fn describe(&self) -> String; }
impl<T: std::fmt::Debug> Describe for T {
    fn describe(&self) -> String { format!("{self:?}") }
}

#[derive(Debug)]                 // теперь Custom: Debug — бланкетная ПОКРЫВАЕТ его
struct Custom;
impl Describe for Custom {       // ОШИБКА E0119: conflicting implementations
    fn describe(&self) -> String { "custom".into() }
}
// То же самое с чужим Debug-типом: impl Describe for String — E0119.
fn main() {}
```

**Разбор.** Два урока сразу. Во-первых, конфликт с бланкетной реализацией возникает, когда тип *фактически* подпадает под её ограничения (`Custom: Debug`), — и добавление `#[derive(Debug)]` внезапно ломает существующий `impl Describe for Custom`. Во-вторых, отсюда правило главы: бланкетный `impl` может писать только крейт, владеющий трейтом, и *добавление* бланкетной реализации в новой версии — ломающее изменение: у нижестоящих крейтов могли быть свои реализации для типов, которые теперь покрыты бланкетной.

## 7. Ограничения трейтов

### Пример 7.1 — ограничение на «весь рецепт», а не на ингредиенты

```rust
use std::collections::HashMap;
use std::hash::BuildHasher;

// Вместо перечисления where K: Hash + Eq, S: BuildHasher + Default —
// одно ограничение, выражающее ИСТИННОЕ требование «это можно собрать из пар».
// Заметьте: в where-клаузе элизия времён жизни не работает — нужен явный 't:
fn count_words<'t, S>(text: &'t str) -> HashMap<&'t str, usize, S>
where
    HashMap<&'t str, usize, S>: FromIterator<(&'t str, usize)>,
    S: BuildHasher + Default,
{
    let mut map = HashMap::default();
    for word in text.split_whitespace() {
        *map.entry(word).or_insert(0) += 1;
    }
    map
}

fn main() {
    let counts: HashMap<&str, usize> = count_words("a b a c a b");
    assert_eq!(counts["a"], 3);
    println!("{counts:?}");
}
```

**Разбор.** Ограничения не обязаны иметь форму `T: Trait` — слева может стоять целый конкретный тип (`HashMap<...>: FromIterator<...>`). Это документирует намерение («мне нужно лишь уметь собрать карту») и автоматически подтягивает правильный набор требований, который не придётся обновлять вслед за std.

### Пример 7.2 — ловушка `#[derive(Clone)]` с `Arc` (из врезки)

```rust
use std::sync::Arc;

struct NotClone;                       // намеренно не Clone

// derive сгенерировал бы: impl<T: Clone> Clone for SharedBad<T> —
// лишнее требование T: Clone, хотя Arc клонируется БЕЗ него!
#[derive(Clone)]
struct SharedBad<T> {
    inner: Arc<T>,
}

struct SharedGood<T> {
    inner: Arc<T>,
}

// Ручная реализация с ПРАВИЛЬНЫМ (пустым) набором ограничений:
impl<T> Clone for SharedGood<T> {
    fn clone(&self) -> Self {
        SharedGood { inner: Arc::clone(&self.inner) }
    }
}

fn main() {
    let good = SharedGood { inner: Arc::new(NotClone) };
    let _copy = good.clone();                          // работает!

    let bad = SharedBad { inner: Arc::new(NotClone) };
    // let _copy = bad.clone();   // ОШИБКА: the trait bound `NotClone: Clone`
    //                            // is not satisfied — хотя клонируем-то Arc!
    let _ = bad;
}
```

**Разбор.** `#[derive(Clone)]` слепо требует `T: Clone` для каждого параметра, даже когда поля клонируются без этого (`Arc<T>`, `Rc<T>`, `&T`, `PhantomData<T>`). Для типов-обёрток над разделяемым состоянием пишите `Clone` вручную — иначе пользователи с не-`Clone` содержимым упрутся в загадочную ошибку.

### Пример 7.3 — HRTB: «для любого времени жизни»

```rust
// Хотим: функция, выбирающая ссылку ВНУТРЬ переданных данных.
// Время жизни результата должно совпадать со временем жизни аргумента —
// для ЛЮБОГО аргумента. Это и есть for<'a>:
fn apply_to_both<F>(f: F, a: &str, b: &str) -> (usize, usize)
where
    F: for<'a> Fn(&'a str) -> &'a str,
{
    (f(a).len(), f(b).len())
}

fn first_word(s: &str) -> &str {
    s.split_whitespace().next().unwrap_or("")
}

fn main() {
    let local_a = String::from("hello brave world");
    let local_b = String::from("rust types");
    // first_word работает с ссылками ЛЮБОГО времени жизни — и local_a, и local_b:
    let (la, lb) = apply_to_both(first_word, &local_a, &local_b);
    assert_eq!((la, lb), (5, 4));
}
```

**Разбор.** Обычный параметр `'a` на функции означал бы «одно конкретное время жизни, выбранное вызывающим». Но замыкание внутри применяется к *двум разным* ссылкам с разными временами жизни — нужно «для всех `'a` сразу»: `for<'a> Fn(&'a str) -> &'a str`. Когда вы пишете просто `F: Fn(&str) -> &str`, компилятор сам подставляет `for<'a>` — поэтому явная форма нужна редко, но именно она работает под капотом.

### Пример 7.4 — листинг 3-6 на практике: `Debug` через HRTB-ограничения

```rust
use std::fmt;

struct Grid<C> {
    cells: C,
}

// Debug для ЛЮБОГО контейнера, который умеет итерироваться по ссылке
// и чьи элементы — Debug. Сам C не обязан быть Debug или Iterator!
impl<C> fmt::Debug for Grid<C>
where
    for<'a> &'a C: IntoIterator,
    for<'a> <&'a C as IntoIterator>::Item: fmt::Debug,
{
    fn fmt(&self, f: &mut fmt::Formatter) -> fmt::Result {
        f.debug_list().entries(&self.cells).finish()
    }
}

fn main() {
    let g1 = Grid { cells: vec![1, 2, 3] };
    let g2 = Grid { cells: std::collections::BTreeSet::from(["b", "a"]) };
    println!("{g1:?}");   // [1, 2, 3]
    println!("{g2:?}");   // ["a", "b"] — работает и для множества!
}
```

**Разбор.** Здесь сразу три приёма из главы: ограничение не на `C`, а на `&'a C`; HRTB `for<'a>` (потому что `fmt` заимствует `self` на своё, неизвестное заранее, время жизни); и обращение к связанному типу через `<&'a C as IntoIterator>::Item`. Один `impl` покрывает `Vec`, `BTreeSet`, `HashMap`, массивы — всё, что итерируется по ссылке.

## 8. Маркерные трейты и маркерные типы

### Пример 8.1 — авто-трейты в действии: один `Rc` портит всю структуру

```rust
use std::rc::Rc;
use std::sync::Arc;

struct Stats {
    counters: Vec<u64>,         // Send + Sync — и Stats автоматически тоже
}

struct BadStats {
    counters: Rc<Vec<u64>>,     // Rc !Send => BadStats АВТОМАТИЧЕСКИ !Send
}

fn main() {
    let s = Stats { counters: vec![1, 2] };
    std::thread::spawn(move || println!("{}", s.counters.len()))
        .join()
        .unwrap();

    let b = BadStats { counters: Rc::new(vec![1, 2]) };
    // std::thread::spawn(move || println!("{}", b.counters.len()));
    // ОШИБКА: `Rc<Vec<u64>>` cannot be sent between threads safely
    // (узнайте Rc по строчке "required because it appears within BadStats")
    let _ = b;

    let fixed = Arc::new(vec![1u64, 2]);     // Arc — атомарный счётчик => Send+Sync
    std::thread::spawn(move || println!("{}", fixed.len())).join().unwrap();
}
```

**Разбор.** `Send`/`Sync` — маркерные авто-трейты: ни одного метода, чистый факт о типе, выводимый рекурсивно из полей. Никто не «вызывает send» — `thread::spawn` лишь *требует* `F: Send + 'static`, и компилятор проверяет предположение, которое в C++ пришлось бы держать в голове. Обратите внимание на качество ошибки: компилятор показывает всю цепочку «почему» вплоть до поля-виновника.

### Пример 8.2 — маркерные типы: состояние соединения в системе типов

```rust
use std::marker::PhantomData;

struct Unauthenticated;     // маркерные типы: ни данных, ни методов,
struct Authenticated;       // только имя-состояние

struct SshConnection<State> {
    host: String,
    _state: PhantomData<State>,     // State не хранится — только помечает
}

impl SshConnection<Unauthenticated> {
    fn connect(host: &str) -> Self {
        SshConnection { host: host.into(), _state: PhantomData }
    }
    // authenticate ПОГЛОЩАЕТ неаутентифицированное соединение
    // и возвращает значение ДРУГОГО типа:
    fn authenticate(self, _password: &str) -> SshConnection<Authenticated> {
        SshConnection { host: self.host, _state: PhantomData }
    }
}

impl SshConnection<Authenticated> {
    // run_command существует ТОЛЬКО у аутентифицированного соединения:
    fn run_command(&self, cmd: &str) -> String {
        format!("[{}] $ {cmd}", self.host)
    }
}

fn main() {
    let conn = SshConnection::connect("example.com");
    // conn.run_command("ls");   // ОШИБКА: нет такого метода у <Unauthenticated> —
    //                           // неправильный порядок вызовов не компилируется!
    let conn = conn.authenticate("hunter2");
    println!("{}", conn.run_command("ls"));
}
```

**Разбор.** Это паттерн *typestate*: состояние объекта закодировано в параметре типа, и методы каждого состояния живут в своём блоке `impl`. Ошибка «выполнить команду до аутентификации» из ошибки времени выполнения превратилась в ошибку компиляции. `PhantomData<State>` нужен, потому что Rust требует, чтобы каждый параметр типа использовался; памяти он не занимает (`size_of` не растёт). Подробнее — в главе 4.

## 9. Экзистенциальные типы

### Пример 9.1 — `impl Trait` в позиции возврата: безымянные типы

```rust
// Тип замыкания невозможно написать руками — у него нет имени.
// impl Trait говорит: "существует некоторый тип, реализующий Fn; поверь":
fn make_adder(x: i32) -> impl Fn(i32) -> i32 {
    move |y| x + y
}

// Цепочка адаптеров без impl Trait имела бы тип
// Filter<Map<RangeInclusive<u32>, {closure}>, {closure}> — и зачем он вам?
fn even_squares(up_to: u32) -> impl Iterator<Item = u32> {
    (1..=up_to).map(|x| x * x).filter(|x| x % 2 == 0)
}

fn main() {
    let add5 = make_adder(5);
    assert_eq!(add5(10), 15);

    let v: Vec<u32> = even_squares(6).collect();
    assert_eq!(v, [4, 16, 36]);
}
```

**Разбор.** В позиции возврата `impl Trait` — экзистенциальный тип: конкретный тип существует и известен компилятору (вывод по телу функции), но в сигнатуре не называется. Вызывающий вправе полагаться лишь на перечисленные трейты (плюс протёкшие авто-трейты вроде `Send` — см. примечание главы). Это и есть стирание типов с нулевой стоимостью: ни `Box`, ни vtable — статическая диспетчеризация сохраняется.

### Пример 9.2 — позиция аргумента ≠ позиция возврата

```rust
use std::fmt::Display;

// В позиции АРГУМЕНТА impl Trait — просто сахар для обобщённого параметра.
// ВЫЗЫВАЮЩИЙ выбирает конкретный тип:
fn announce(item: impl Display) {
    println!("--> {item}");
}
// то же самое, что fn announce<T: Display>(item: T)

// В позиции ВОЗВРАТА конкретный тип выбирает ФУНКЦИЯ (и он один!):
fn pick(flag: bool) -> impl Display {
    // if flag { 42 } else { "str" }   // ОШИБКА: два РАЗНЫХ типа —
    //                                 // экзистенциальный тип должен быть одним
    if flag { 42 } else { 7 }          // ок: оба плеча — i32
}

// Нужны действительно разные типы под одной сигнатурой? Это работа dyn:
fn pick_dyn(flag: bool) -> Box<dyn Display> {
    if flag { Box::new(42) } else { Box::new("str") }
}

fn main() {
    announce(3.14);
    announce("hello");
    println!("{} {}", pick(true), pick_dyn(false));
}
```

**Разбор.** Ключевая асимметрия: `impl Trait`-аргумент — «любой тип, какой даст вызывающий» (универсальный, мономорфизация); `impl Trait`-возврат — «один конкретный тип, какой выбрало тело» (экзистенциальный). Если из разных веток нужно вернуть разные типы — это уже динамика, берите `Box<dyn Trait>`.

### Пример 9.3 — стирание типов скрывает детали реализации

```rust
mod library {
    // ВНУТРИ: сложный конкретный тип итератора.
    // СНАРУЖИ: только "что-то, выдающее (имя, балл)".
    pub fn top_scores(raw: &str) -> impl Iterator<Item = (&str, u32)> + '_ {
        raw.lines()
            .filter_map(|l| l.split_once(':'))
            .filter_map(|(name, s)| Some((name.trim(), s.trim().parse().ok()?)))
            .filter(|&(_, score)| score >= 50)
    }
    // Завтра можно переписать на свой struct TopScores<'a> { ... } —
    // сигнатура (и пользователи) не изменятся: тип и так был скрыт.
}

fn main() {
    let data = "alice: 80\nbob: 42\ncarol: 95";
    for (name, score) in library::top_scores(data) {
        println!("{name}: {score}");
    }
    // alice: 80
    // carol: 95
}
```

**Разбор.** Если бы функция возвращала именованную цепочку `FilterMap<FilterMap<Lines, ...>>`, эти служебные типы стали бы частью публичного API — любая правка тела ломала бы пользователей. `impl Trait` оставляет конкретный тип деталью реализации. Заметьте `+ '_`: возвращаемый итератор заимствует `raw`, и это нужно отразить в сигнатуре.

## Шпаргалка по главе

- **Выравнивание**: каждый тип кладётся по адресу, кратному своему выравниванию; у составного типа выравнивание = max по полям; невыровненный доступ — медленно или крах.
- **Раскладка**: `repr(C)` — поля по порядку + набивка (для FFI и unsafe-кастов); `repr(Rust)` — компилятор волен переупорядочивать (и гарантий нет даже для «одинаковых» типов); `repr(packed)` — без набивки, но `&поле` запрещён; `repr(align(n))` — увеличить выравнивание (линии кэша); `repr(transparent)` — раскладка единственного поля (newtype).
- **Enum** = union вариантов + дискриминант; нишевая оптимизация прячет дискриминант в недопустимые битовые комбинации (`Option<&T>` не дороже `&T`).
- **DST** (`str`, `[T]`, `dyn Trait`) не имеют размера на этапе компиляции — живут только за **толстым указателем** (2 слова: адрес + длина или адрес + vtable); `T: ?Sized` снимает неявное `T: Sized`.
- **Статическая диспетчеризация** (обобщения): мономорфизация — копия кода на тип, быстро, но раздувает бинарник; для общих частей — приём «необобщённая внутренняя функция». **Динамическая** (`dyn Trait`): одна копия + vtable, гетерогенные коллекции, чуть медленнее. Библиотеки → static (пользователь сам решит), бинарники → часто dyn.
- **Объектная безопасность**: в трейт-объект нельзя — методы с `Self` в сигнатуре, обобщённые методы, статические методы; `where Self: Sized` на методе исключает его из dyn-интерфейса.
- **Связанный тип** — одна реализация на тип, ноль аннотаций (`Iterator::Item`); **параметр трейта** — много реализаций, нужна дизамбигуация (`From<T>`). По умолчанию — связанный тип.
- **Когерентность**: одна реализация на пару (тип, трейт) во всей программе. **Правило сироты**: трейт или тип должен быть локальным; послабления — фундаментальные типы (`&`, `&mut`, `Box`) и покрытые реализации (локальный тип раньше обобщённых параметров). Бланкетный impl — только у владельца трейта; его добавление — ломающее изменение. Обход — newtype.
- **Ограничения**: слева может стоять любой тип (`HashMap<T, usize, S>: FromIterator<...>`); связанные типы — через `I::Item` / `<T as Trait>::Item`; «для любого времени жизни» — HRTB `for<'a>`; derive дописывает `T: Trait` даже там, где не надо (`Arc<T>`) — иногда `impl` пишут вручную.
- **Маркерные трейты** (`Send`, `Sync`, `Sized`, `Unpin`) — факты о типе без методов; большинство — авто-трейты (выводятся из полей). **Маркерные типы** (пустые структуры) + `PhantomData` = typestate: недопустимые переходы состояний не компилируются.
- **Экзистенциальные типы**: `impl Trait` в возврате — «существует один конкретный тип» (стирание типов без аллокаций и vtable); в аргументе — сахар для обобщённого параметра; разные типы из разных веток — это уже `Box<dyn Trait>`.
